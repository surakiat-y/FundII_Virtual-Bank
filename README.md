## รายละเอียดโดยย่อ


**ตัวอย่างการเขียน (โปรแกรม)**

โครงงานนี้เป็นระบบธนาคารในรูปแบบ web application โดยมีฟังก์ชันพื้นฐานของระบบธนาคาร คือ ฝากเงิน โอนเงิน ตรวจสอบยอดเงิน และมีการตรวจสอบความผิดปกติของการใช้งาน สามารถที่จะแบ่งบัญชีย่อยได้ มีระบบแอดมินสามารถระงับบัญชีได้ โดยโปรเจคนี้เป็นเพียงการจำลองฟังค์ชันพื้นฐานของระบบธนาคารเท่านั้น


## คุณลักษณะและขอบเขต

### คุณลักษณะขั้นต่ำ
ระบบจัดการบัญชี: สามารถสมัครสมาชิก (Register) และเข้าสู่ระบบ (Login) โดยมีการเก็บรหัสผ่านแบบเข้ารหัส (Hashing)

ฟังก์ชันธุรกรรมหลัก:

Deposit/Withdraw: ฝากเงินและถอนเงินออกจากบัญชีหลัก

Transfer: โอนเงินระหว่างบัญชีภายในระบบโดยใช้เลขบัญชี

ระบบบัญชีย่อย (Sub-Accounts): ผู้ใช้สามารถสร้าง "กระเป๋าเงินย่อย" (เช่น เงินออม, ค่าใช้จ่าย) ภายใต้บัญชีหลักเดียวได้

หน้าจอแสดงผล (GUI): มี Dashboard แสดงยอดเงินรวม ยอดเงินในบัญชีย่อย และประวัติการทำรายการล่าสุด

การจัดการข้อมูล: บันทึกข้อมูลผู้ใช้ ยอดเงิน และประวัติธุรกรรมลงในไฟล์ JSON หรือฐานข้อมูล SQLite


### คุณลักษณะเพิ่มเติม

ระบุส่วนที่คิดจะทำเพิ่มเติมเมื่อมีเวลาพอ เช่น

Admin Dashboard: หน้าจอพิเศษสำหรับแอดมินเพื่อตรวจสอบธุรกรรมทั้งหมด และสามารถ "ระงับบัญชี" (Suspend) หากพบพฤติกรรมน่าสงสัย

Basic Anomaly Detection: ระบบแจ้งเตือนเมื่อมีการโอนเงินเกินวงเงินที่กำหนด หรือมีการ Login ผิดพลาดติดต่อกันหลายครั้ง (Brute-force protection)

Export Statement: สามารถส่งออกประวัติการทำรายการเป็นไฟล์ CSV หรือ PDF ได้

## การแบ่งงาน

ระบุแผนการแบ่งความรับผิดชอบของสมาชิกแต่ละคนในการทำโครงงาน ทั้งนี้ สมาชิกทุกคนต้องมีส่วนร่วมในการเขียนโค้ด เช่น

- **ชื่อสมาชิกคนที่ 1**: นายนันทพัฒน์ วรรณลักษณ์ (สมาขิกคนที่ 1)
ออกแบบและเขียนโค้ดส่วน Business Logic (การคำนวณเงิน, ระบบโอนเงิน)

พัฒนาระบบความปลอดภัย (Hashing Password, Anomaly Detection Logic)

จัดการการบันทึกข้อมูล (Database/JSON Handling)

- **ชื่อสมาชิกคนที่ 2**: นายสุรเกียรติ์ ยิ้มสมบูรณ์ (สมาชิกคนที่ 2)
ออกแบบและพัฒนา GUI ทั้งหมด (Login, Dashboard, Admin Page)

จัดการส่วน User Input Validation (ตรวจสอบความถูกต้องของข้อมูลที่กรอก)

ทำเอกสารสรุปโครงการและออกแบบคู่มือการใช้งาน	

## ความท้าทายและความเสี่ยง

ความท้าทายด้านความถูกต้องของข้อมูล (Data Integrity):

ปัญหา: ยอดเงินอาจคลาดเคลื่อนหากระบบล่มระหว่างทำธุรกรรม

แนวทางแก้ไข: ศึกษาเรื่อง Transaction Management หรือการใช้ synchronized ใน Java เพื่อจัดการการเข้าถึงข้อมูลพร้อมกัน

ความท้าทายด้านการจัดการ GUI:

ปัญหา: การสร้าง Dashboard ที่ซับซ้อนด้วย Java Swing อาจใช้เวลานาน

แนวทางแก้ไข: เลือกใช้ JavaFX หรือศึกษาการใช้ Layout Manager ให้แม่นยำ หรือใช้เครื่องมือช่วยออกแบบเช่น Scene Builder

ความเสี่ยงด้านการจัดเก็บข้อมูล:

ปัญหา: หากเก็บข้อมูลในไฟล์ธรรมดา ผู้ใช้อาจแอบเปิดไฟล์ไปแก้ตัวเลขยอดเงินเองได้

แนวทางแก้ไข: ศึกษาการใช้ Checksum หรือ Simple Encryption เพื่อตรวจสอบว่าไฟล์ข้อมูลถูกแก้ไขภายนอกหรือไม่
